# 2인·1주 MVP 개발 계획

## 1. 개발 목표

### 최종 사용자 시나리오

일주일 뒤 다음 흐름이 실제로 동작하는 것을 목표로 한다.

```text
웹 로그인
→ 조각글·사진·링크 저장
→ 수신함에서 글감 확인
→ 프로젝트 생성
→ 글감을 프로젝트에 연결
→ 연결된 글감을 보면서 문서 작성
→ 모바일에서 같은 계정으로 조각글·사진·링크 추가
→ 웹 수신함에 즉시 표시
```

이번 개발의 목표는 완성된 상용 서비스가 아니라 다음을 검증할 수 있는 **작동하는 프로토타입**이다.

1. 사용자가 모바일과 웹에서 글감을 지속해서 수집하는가?
2. 수집한 글감을 실제 글쓰기에 연결하는가?
3. 모바일 수집과 웹 글쓰기 역할 분리가 자연스러운가?

# 2. 일주일 MVP 범위

## 2.1 반드시 구현하는 기능

### 웹

| 화면    | 기능                       |
| ----- | ------------------------ |
| 로그인   | Google 소셜 로그인            |
| 메인    | 최근 글감, 수신함 개수, 진행 중 프로젝트 |
| 글감 기록 | 조각글, 사진, URL             |
| 글감함   | 전체 글감 목록, 유형 필터          |
| 글감 상세 | 내용 확인, 수정, 삭제, 프로젝트 연결   |
| 프로젝트  | 생성, 목록, 상세               |
| 글쓰기   | 제목, 본문, 자동 저장, 연결된 글감 확인 |
| 프로필   | 사용자 이름, 로그아웃             |

### 모바일

| 화면    | 기능                 |
| ----- | ------------------ |
| 로그인   | 웹과 동일한 계정          |
| 홈     | 최근 글감 목록           |
| 글감 기록 | 조각글, 사진 촬영·선택, URL |
| 글감 상세 | 저장한 내용 확인          |
| 프로필   | 계정 확인, 로그아웃        |

모바일에서는 프로젝트 관리와 글쓰기를 지원하지 않는다.

```text
모바일 = 글감 수집
웹 = 글감 정리와 글쓰기
```

# 3. 기술 구성

## 3.1 웹

```text
Next.js App Router
TypeScript
Tailwind CSS
Supabase JavaScript Client
Vercel
```

Next.js App Router는 파일 시스템 기반 라우팅을 사용하므로 로그인, 수신함, 프로젝트, 문서 화면을 디렉터리 구조와 일치시켜 빠르게 구성하기 적합하다.

## 3.2 모바일

```text
Flutter
Dart
go_router
image_picker
supabase_flutter
http
```

Flutter는 Android와 iOS에서 동일한 Dart UI 및 비즈니스 로직을 공유한다. `go_router`로 선언적 라우팅을 구성하고 `image_picker`로 시스템 사진 선택과 카메라 촬영 진입을 제공한다. 인증은 `supabase_flutter`, REST API 호출은 Dart HTTP 클라이언트를 사용한다.

## 3.3 데이터와 인증

```text
Supabase Auth
Supabase PostgreSQL
Supabase Storage
Row Level Security
```

Supabase Auth와 PostgreSQL RLS를 결합하면 웹과 모바일 클라이언트가 데이터베이스에 직접 접근하면서도 `user_id = auth.uid()` 조건으로 사용자별 데이터를 제한할 수 있다. Storage도 RLS 기반 접근 정책을 적용할 수 있다.

## 3.4 글쓰기 편집기

MVP에서는 Tiptap 대신 일반 `<textarea>`를 사용한다.

지원 기능은 다음으로 제한한다.

* 제목
* 일반 텍스트 본문
* 자동 저장
* 글자 수
* 연결된 글감 패널

Tiptap은 커스텀 리치 텍스트 편집기를 구현할 수 있는 headless 프레임워크지만, 일주일 MVP에서는 편집기 설정과 예외 처리가 일정 위험 요소가 된다. 웹 핵심 기능이 일찍 완료된 경우에만 StarterKit 적용을 검토한다.

# 4. API 전략 변경

기존 기획처럼 별도의 REST API 전체를 만들지 않는다.

## 4.1 클라이언트에서 직접 처리

다음 기능은 웹과 모바일에서 Supabase Client를 통해 직접 처리한다.

* 로그인
* 글감 CRUD
* 프로젝트 CRUD
* 글감과 프로젝트 연결
* 문서 CRUD
* 사진 업로드
* 프로필 조회

```text
웹·모바일
→ Supabase Auth
→ RLS가 적용된 PostgreSQL 및 Storage
```

이 방식은 별도의 API 컨트롤러, 인증 미들웨어, 요청·응답 DTO를 중복 개발하지 않아도 된다는 장점이 있다.

## 4.2 별도 서버 처리가 필요한 기능

Next.js Route Handler는 링크 미리보기 하나만 만든다.

```http
POST /api/link-preview
```

요청:

```json
{
  "url": "https://example.com/article"
}
```

응답:

```json
{
  "title": "페이지 제목",
  "description": "페이지 설명",
  "imageUrl": "https://example.com/image.jpg",
  "siteName": "Example",
  "canonicalUrl": "https://example.com/article"
}
```

실패 응답:

```json
{
  "title": null,
  "description": null,
  "imageUrl": null,
  "siteName": null,
  "canonicalUrl": "https://example.com/article"
}
```

링크 미리보기 처리는 3초 정도의 제한을 두고, 실패하더라도 글감 저장을 막지 않는다.

# 5. 최소 DB 스키마

기존 스키마에서 워크스페이스, 태그, 컬렉션, 작업 큐, 문서 버전 등을 제거한다.

## 5.1 관계 구조

```text
auth.users
  └─ profiles

auth.users
  ├─ captures
  │   └─ capture_assets
  │
  ├─ projects
  │   ├─ project_captures ─ captures
  │   └─ documents
```

## 5.2 `profiles`

```sql
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  display_name text,
  avatar_url text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

## 5.3 `captures`

```sql
create type public.capture_type as enum (
  'text',
  'image',
  'link'
);

create table public.captures (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  type public.capture_type not null,
  title text,
  body_text text,
  source_url text,
  source_metadata jsonb not null default '{}'::jsonb,
  captured_at timestamptz not null default now(),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

`source_metadata` 예시:

```json
{
  "title": "외부 페이지 제목",
  "description": "외부 페이지 설명",
  "imageUrl": "https://example.com/image.jpg",
  "siteName": "Example"
}
```

## 5.4 `capture_assets`

```sql
create table public.capture_assets (
  id uuid primary key default gen_random_uuid(),
  capture_id uuid not null
    references public.captures(id)
    on delete cascade,
  user_id uuid not null
    references auth.users(id)
    on delete cascade,
  storage_path text not null,
  mime_type text not null,
  size_bytes bigint,
  width integer,
  height integer,
  created_at timestamptz not null default now()
);
```

MVP에서는 하나의 글감에 사진 한 장만 허용한다. DB 구조는 여러 장을 지원하지만 UI에서는 한 장만 선택하게 한다.

## 5.5 `projects`

```sql
create table public.projects (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  title text not null,
  description text,
  status text not null default 'active'
    check (status in ('active', 'completed', 'archived')),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

## 5.6 `project_captures`

```sql
create table public.project_captures (
  project_id uuid not null
    references public.projects(id)
    on delete cascade,
  capture_id uuid not null
    references public.captures(id)
    on delete cascade,
  user_id uuid not null
    references auth.users(id)
    on delete cascade,
  created_at timestamptz not null default now(),
  primary key (project_id, capture_id)
);
```

## 5.7 `documents`

MVP에서는 프로젝트당 문서를 하나만 허용한다.

```sql
create table public.documents (
  id uuid primary key default gen_random_uuid(),
  project_id uuid not null unique
    references public.projects(id)
    on delete cascade,
  user_id uuid not null
    references auth.users(id)
    on delete cascade,
  title text not null default '',
  content_text text not null default '',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

## 5.8 핵심 RLS 정책

모든 테이블에서 다음 원칙을 사용한다.

```sql
alter table public.captures enable row level security;

create policy "users can select own captures"
on public.captures
for select
using (auth.uid() = user_id);

create policy "users can insert own captures"
on public.captures
for insert
with check (auth.uid() = user_id);

create policy "users can update own captures"
on public.captures
for update
using (auth.uid() = user_id)
with check (auth.uid() = user_id);

create policy "users can delete own captures"
on public.captures
for delete
using (auth.uid() = user_id);
```

같은 형태를 다음 테이블에 적용한다.

* `profiles`
* `capture_assets`
* `projects`
* `project_captures`
* `documents`

# 6. 저장소 구조

단일 Git 저장소에서 웹, 모바일, 공통 타입, Supabase 마이그레이션을 관리한다.

```text
inspiration-app/
├─ apps/
│  ├─ web/
│  │  ├─ app/
│  │  ├─ components/
│  │  ├─ features/
│  │  └─ lib/
│  │
│  └─ mobile/
│     ├─ app/
│     ├─ components/
│     ├─ features/
│     └─ lib/
│
├─ packages/
│  └─ shared/
│     ├─ types/
│     ├─ schemas/
│     └─ constants/
│
├─ supabase/
│  ├─ migrations/
│  └─ seed.sql
│
├─ .env.example
├─ package.json
└─ pnpm-workspace.yaml
```

## 공통 패키지에 포함할 것

```text
Capture 타입
Project 타입
Document 타입
입력 검증 스키마
날짜 포맷 함수
글감 유형 상수
Supabase DB 생성 타입
```

공통 UI 컴포넌트는 공유하지 않는다.

웹과 모바일은 표현 방식이 다르기 때문에 다음만 공유한다.

```text

검증 규칙
데이터 접근 함수 일부
상수
```

# 7. 화면 구조

## 7.1 웹 라우트

```text
/login
/
/captures
/captures/new
/captures/[captureId]
/projects
/projects/new
/projects/[projectId]
/projects/[projectId]/write
/profile
```

### `/`

* 빠른 글감 기록 버튼
* 최근 글감 6개
* 수신함 전체 보기
* 프로젝트 3개
* 새 프로젝트 버튼

### `/captures`

* 전체 글감 목록
* 유형 필터: 전체, 조각글, 사진, 링크
* 최신순 정렬
* 새 글감 버튼

검색은 구현 시간이 남을 때만 추가한다.

### `/projects/[projectId]/write`

```text
┌──────────────┬──────────────────────┬─────────────────┐
│ 프로젝트     │ 문서                 │ 연결된 글감     │
│ 정보         │                      │                 │
│              │ 제목                 │ 글감 카드       │
│ 뒤로 가기    │ 본문 textarea        │ 원문 보기       │
│              │ 저장 상태            │                 │
└──────────────┴──────────────────────┴─────────────────┘
```

화면이 좁은 경우 오른쪽 글감 패널은 버튼으로 열리는 드로어로 변경한다.

## 7.2 모바일 라우트

```text
/login
/(tabs)/home
/(tabs)/profile
/capture
/capture/text
/capture/image
/capture/link
/captures/[captureId]
```

하단 탭은 두 개만 둔다.

```text
홈
프로필
```

글감 기록은 중앙 플로팅 `+` 버튼으로 진입한다.

# 11. Git 작업 규칙

## 브랜치

```text
main
feat/auth
feat/capture-create
feat/capture-list
feat/project
feat/document-editor
feat/mobile-capture

```

## 원칙

* 하루 이상 유지되는 대형 브랜치를 만들지 않는다.
* 기능 단위로 짧게 작업한다.
* PR은 당일 병합한다.

* 개발자 B가 SQL과 RLS를 반드시 검토한다.
* 병합 전 lint와 typecheck를 통과한다.
* `main`은 항상 배포 가능한 상태로 유지한다.

## PR 설명

```text
무엇을 변경했는가
어떻게 확인하는가
DB 변경이 있는가
환경 변수 변경이 있는가
알려진 제한은 무엇인가
```

# 16. 가장 위험한 부분

## 16.1 모바일 OAuth

웹 OAuth보다 리다이렉트와 딥링크 설정에서 문제가 발생할 가능성이 높다.

대응:

* Day 4 오전에 가장 먼저 실제 기기로 테스트
* 실패 시 이메일 OTP로 대체
* 마지막 날까지 미루지 않기

## 16.2 사진 업로드와 Storage RLS

DB 레코드는 생성됐지만 사진 업로드가 거부되거나, 반대로 다른 사용자의 사진 경로에 접근하는 문제가 발생할 수 있다.

대응:

* Day 2에 실제 사용자 두 명으로 테스트
* 파일 경로에 사용자 ID 포함
* 비공개 버킷 사용
* 서명된 URL 또는 인증된 다운로드 사용
* DB RLS와 Storage 정책을 별도로 검증

## 16.3 글쓰기 편집기

리치 텍스트, 드래그 앤 드롭, 참조 블록 등을 추가하기 시작하면 프로젝트 전체 일정을 초과할 수 있다.

대응:

```text
제목 input
본문 textarea
연결 글감 읽기 전용 패널
```

이 구조를 유지한다.

## 16.4 링크 미리보기

사이트별 차단, 잘못된 HTML, 리다이렉트, 느린 응답 등이 발생할 수 있다.

대응:

* 미리보기는 부가 기능으로 취급
* 실패해도 URL 저장
* 인스타그램·유튜브 전용 파서 미구현
* 짧은 요청 제한 설정

# 17. MVP 성공 기준

일주일 뒤 아래 조건을 만족하면 성공으로 판단한다.

## 기능 기준

* 웹 전체 핵심 흐름 성공
* 모바일 수집 흐름 성공
* 두 환경에서 동일 데이터 확인
* 사용자별 데이터 격리
* 사진 업로드 성공
* 문서 자동 저장 성공

## 사용자 테스트 기준

최소 3명의 테스트 사용자가 설명 없이 다음 작업을 수행한다.

```text
글감 하나 저장
프로젝트 하나 생성
글감을 프로젝트에 연결
글 한 문단 작성
모바일에서 사진 글감 추가
```

다음 질문에 답하게 한다.

1. 글감 저장 버튼을 쉽게 찾았는가?
2. 수신함과 프로젝트의 차이를 이해했는가?
3. 모바일에서 저장한 자료를 웹에서 찾을 수 있었는가?
4. 연결된 글감을 보면서 글을 쓰는 방식이 유용했는가?
5. 다시 사용할 의향이 있는가?

# 18. 개발 이후 첫 번째 우선순위

MVP 이후에는 기능을 바로 추가하지 않고 사용 기록을 먼저 확인한다.

우선순위 후보는 다음 순서로 평가한다.

```text
1. 외부 앱 공유 기능
2. 여러 사진 첨부
3. 태그
4. 캘린더 보기
5. 컬렉션
6. 리치 텍스트 편집기
7. 유튜브·영화 메타데이터
8. AI 분류와 관련 글감 추천
```

사용자가 모바일에서 URL을 자주 수집한다면 공유 확장을 먼저 개발한다. 사용자가 수집은 많이 하지만 글감을 찾지 못한다면 태그와 검색을 먼저 개발한다. 글쓰기 전환율이 낮다면 프로젝트와 편집기 경험을 먼저 개선한다.
